# WhatsApp for your Claude — self-hosted

A Claude assistant that can read, summarize, draft, and send WhatsApp messages on your behalf. The whole thing runs on your laptop. Your messages, your token, your WhatsApp session — none of it leaves your machine.

You talk to Claude. Claude talks to a small backend running locally. The backend talks to WAHA Plus (also running locally) which talks to WhatsApp. Three Docker-ish components, one `docker compose up`, one MCP connector in Claude.

---

## Quick start (≈ 3 minutes)

You need:
- **Docker Desktop** ([download](https://www.docker.com/products/docker-desktop/)) — running.
- A **WAHA Plus access token** from [waha.devlike.pro](https://waha.devlike.pro/) (this is the only thing that costs money — it gates the WhatsApp gateway image).
- **Node.js 18+** ([download](https://nodejs.org/)) — only for the one-shot setup script.
- A **Claude client** that supports HTTP MCP servers: Claude Code (easiest), Claude Desktop, or Claude.ai web (needs a public tunnel — see below).

Then:

```bash
git clone https://github.com/arnav-chauhan-kgpian/waha-claude.git
cd waha-claude

docker login                     # username: devlikeapro, password: your WAHA Plus token
node setup.mjs                   # or: ./setup.sh (Unix) / setup.cmd (Windows)
```

`setup.mjs` will:
1. Generate strong random keys in `.env`.
2. `docker compose up -d --build` — pulls WAHA, builds the backend.
3. Wait for it to come up, sign you up, mint your bearer token.
4. Write `.mcp.json` so Claude Code picks it up automatically.
5. Print copy-paste config for Claude Desktop / Claude.ai web.

Then in **Claude Code** (from this directory):

```
claude
> Connect my WhatsApp
```

Claude shows a QR inline. You scan with WhatsApp → Settings → Linked devices → Link a device. Done.

## Using it

Once connected, ask Claude things like:

| Say to Claude | What happens |
|---|---|
| "Connect my WhatsApp" | QR shows inline; scan to link. |
| "Am I connected?" | Reports state + your phone number. |
| "List my recent chats" | Shows recent chats with names, unread counts. |
| "Summarize my unread WhatsApp" | Reads unread chats, categorizes them. |
| "Reply to <name> that …" | Drafts a reply, **asks for confirmation**, then sends. |
| "Read my chat with <name>" | Pulls recent messages from one chat. |
| "Disconnect WhatsApp" | Stops the session. |

Claude **never auto-sends**. Every send requires you to say "yes" / "send" / "ok send" in the same turn.

## Tools the user's Claude sees

| Tool | What it does |
|---|---|
| `connect_whatsapp` | Create or refresh the WhatsApp session; render QR inline. |
| `get_qr` | Re-fetch the QR for re-scanning. |
| `whatsapp_status` | `SCAN_QR_CODE` / `WORKING` / `STARTING` / `STOPPED` / `FAILED` / `NEW`. |
| `list_chats` | Recent chats with contact names stitched from your address book. |
| `get_messages` | Messages in a chat with author resolved (group members named where possible). |
| `summarize_unread` | Unread snippets categorized into Action Required / Awaiting Replies / Important Updates / Low Priority. |
| `draft_reply` | History + your instruction so Claude can compose a draft. **Does not send.** |
| `send_message` | Send. Rate-limited (30/min, env-tunable). Recorded in audit log. Claude must confirm with you first. |
| `disconnect_whatsapp` | Stop the session. |

`summarize_unread` and `draft_reply` are **context-only**: they return raw data; the LLM work runs on your Claude. The backend never calls any LLM API, so there's no Anthropic API key to manage.

## What runs where

```
Your Claude  ─MCP/http─►  Backend (Node)  ─http─►  WAHA Plus  ─►  WhatsApp
                          ↳ SQLite at /app/data
```

Both Docker containers live in this repo's `docker-compose.yml`. WAHA is bound to `127.0.0.1:3000`, the backend to `127.0.0.1:8080` — nothing exposed to your network unless you tunnel.

## Privacy

- Your WhatsApp session, your message history, and your audit log live in **Docker volumes on your laptop only**. Nothing is sent anywhere except WhatsApp itself.
- Your MCP bearer token is stored only in `.mcp.json` (and as SHA-256 in the backend's SQLite). It never leaves this machine.
- The signup `INVITE_CODE` and `WAHA_API_KEY` in `.env` are generated by `setup.mjs` — they never need to be the same as anyone else's.

## Useful commands

```bash
docker compose logs -f backend          # tail backend logs
docker compose logs -f waha             # tail WAHA logs
docker compose ps                       # what's running
docker compose down                     # stop, keep data
docker compose down -v                  # stop AND wipe everything (you'll re-link)
```

## Letting Claude.ai web reach this

`localhost` isn't reachable from claude.ai. If you need web access (instead of Claude Code or Desktop), expose your local backend with a free tunnel:

```bash
cloudflared tunnel --url http://localhost:8080
```

Copy the printed `https://<random>.trycloudflare.com` URL. In Claude.ai → Settings → Connectors → Add custom connector:
- URL: `https://<random>.trycloudflare.com/mcp`
- Header: `Authorization: Bearer <your-token-from-.mcp.json>`

## Going multi-tenant (operator-hosted)

The same code can run as a multi-user service in a cluster — one shared backend, many users with their own tokens. See [HANDOFF.md](HANDOFF.md) for the AKS deployment + the [k8s/](k8s/) manifests, and [DESIGN.md](DESIGN.md) for the full architecture, schema, and isolation model.
